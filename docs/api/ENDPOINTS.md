## Key Headers

### Request Headers
- `Authorization: Bearer <token>`: Required for authenticated endpoints.
- `X-API-Key: <key>`: Required for internal/webhook routes.
- `X-Correlation-Id: <uuid>`: Optional. Client-provided ID for request tracing. If not provided, the server generates one.
- `Content-Type: application/json`: Required for POST/PUT/PATCH requests with body.

### Response Headers
- `X-Correlation-Id: <uuid>`: The correlation ID for this request (either provided by client or generated by server). Use this to trace the request in logs.

## Error Response Format

All errors follow a standardized format:

```json
{
  "timestamp": "2026-04-29T12:00:00Z",
  "status": 400,
  "code": "VALIDATION_ERROR",
  "message": "Validation failed",
  "path": "/api/v1/operations/entries",
  "correlationId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "details": {
    "plate": "Plate cannot be blank"
  }
}
```

### Error Codes
- `VALIDATION_ERROR`: Request validation failed (400)
- `OPERATION_ERROR`: Business logic error (various status codes)
- `INVALID_ARGUMENT`: Invalid parameter value (400)
- `INTERNAL_ERROR`: Unexpected server error (500)
- `UNAUTHORIZED`: Authentication required (401)
- `FORBIDDEN`: Insufficient permissions (403)

## Base URL
Local Development: `http://localhost:8080`

## Tracing Requests

To debug any API issue:
1. Include or capture the `X-Correlation-Id` header from the response
2. Search logs using: `grep "<correlation-id>" apps/api/logs/parkflow.log`
3. Follow the [Debug Runbook](../runbooks/debug-api-request.md) for detailed steps
